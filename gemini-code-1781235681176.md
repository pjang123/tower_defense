# Tower Defense: Code Review & Optimization Report

Based on a review of the core Java files in the `tower_defense` repository, the following areas have been identified for improvement, optimization, and refactoring.

---

## 1. Performance & Server Lag (TPS) Optimizations

### Synchronous World Cloning (`GameManager.java`)
In `startMatch()`, the plugin uses `copyDirectory` to physically clone world folders (region files, `level.dat`, etc.) entirely on the main server thread. Because file I/O is blocking, doing this synchronously will cause a massive lag spike (TPS drop) every time a match starts.
* **Optimization:** Run the file I/O operations asynchronously using `Bukkit.getScheduler().runTaskAsynchronously()`. Once the files are successfully copied, fire a synchronous callback to the main thread to execute `Bukkit.createWorld()` and teleport the players.

### Heavy Ticker Operations (`TowerManager.java`)
The `startTowerTicker()` runs every single tick (50ms), creating multiple performance bottlenecks:
* **O(N²) Distance Checks:** For `TowerType.REDSTONE`, the loop iterates through *every* placed tower and checks `distanceSquared` to apply boosts. If players place 50 towers, this evaluates hundreds of times a second.
* **Costly Spatial Queries:** `getMobsInRadius()` is called frequently by Golem, Beehive, and AoE towers. It iterates over *all active mobs* in the match to calculate distances. This is an O(T × M) operation (Towers × Mobs).
* **Optimization:** Cache targets for a few ticks rather than scanning every tick. Implement spatial partitioning (like bucketing mobs into their current chunk or path segment) so towers only measure distance against mobs physically nearby.

### Expensive Entity Lookups (`TowerManager.java`)
In `tickDripstoneHazards()`, the plugin checks for the hazard's `BlockDisplay` using `loc.getWorld().getNearbyEntities(...)`. Doing bounding-box entity searches in a ticker is computationally expensive.
* **Optimization:** Instead of storing a `List<Location> hazardTiles`, store a wrapper object that holds a direct reference to the `BlockDisplay` entity, bypassing the need to search the world for it every tick.

### Reflection Overhead
When placing a `HappyGhast` tower, the code uses reflection to find the `setHarnessed` method (`ghast.getClass().getMethod(...)`). This method lookup happens *every time* the tower is placed.
* **Optimization:** Cache the `Method` object in a static variable at class load time so the expensive lookup is only performed once.

## 2. Architecture & OOP (Code Smells)

### Data Fragmentation (`GameManager.java`)
Player data is scattered across roughly 10 different HashMaps (`playerGold`, `playerExp`, `playerUnlockedMobs`, `goldGenLevels`, `swordLevels`, `bowLevels`, `playerArenas`, `playerMobTiers`, etc.).
* **Improvement:** This violates the principle of high cohesion. Wrap these into a single `PlayerSession` or `TDPlayer` class, and store them in a single `Map<UUID, PlayerSession>`. This makes memory cleanup much easier and prevents memory leaks.

### Telescoping Methods (`MobManager.java`)
There are 8+ overloaded versions of `spawnMob()` passing an increasingly long list of parameters (speed, health, armor, immunities, rewards, etc.).
* **Improvement:** Replace these overloads with a **Builder Pattern** (e.g., `new MobBuilder(type).speed(2.0).health(100).build()`) or pass a configuration POJO. It will vastly improve code readability and maintainability.

### Blatant Code Duplication
In `MobManager.java`, the `summonZombieAt` method admits in a comment that it couldn't reuse `spawnMobByChain` due to teleporting bugs, so it completely re-implements ~50 lines of mob setup logic (collisions, PDC tags, AI wiping).
* **Improvement:** Add a `Location spawnLoc` override to the base `spawnMob` method. Reusing the centralized setup logic prevents bugs if you ever decide to update how TD mobs spawn globally.

## 3. Safety & Bukkit API Practices

### Inventory Title Matching (`MobListener.java`)
Menu clicks are intercepted by matching the exact title string of the inventory (e.g., `if (title.equals(ChatColor.DARK_BLUE + "Open Games"))`). This is extremely brittle; if anyone changes the color code or text during UI creation, the click listener instantly breaks.
* **Improvement:** Use a custom `InventoryHolder` (e.g., `public class MenuHolder implements InventoryHolder`) and check `if (event.getInventory().getHolder() instanceof MenuHolder)`.

### Particle Flooding (`GameManager.java`)
The Spell Particle Ticker runs every 5 ticks, iterating over *all* waypoints in an arena and spawning particles if a spell (like FREEZE or OVERCHARGE) is active. On a map with 200 waypoints, this creates 800 particle packets per second per arena, which can throttle client rendering and network bandwidth.
* **Improvement:** Only spawn particles around the leading mobs, or increase the interval between particle spawns.

### Magic Strings for State
The code uses `"1"` and `"2"` as strings everywhere to denote the two arenas/teams.
* **Improvement:** Upgrade this to a proper `Team` or `Arena` Enum. It avoids arbitrary bugs caused by string mismatches (e.g., `mobArena.equals("1")`) and makes the code self-documenting.
